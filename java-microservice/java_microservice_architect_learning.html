Java Microservice Architecture

A complete guide from monolith to distributed systems — designed for Java engineers.

What are Microservices?
Small, independently deployable services that do one thing well. Each owns its data and communicates via APIs.
Vs. Monolith
Monoliths are single deployable units. Microservices break them into bounded contexts, enabling team autonomy and scaling.
Why Java?
Spring Boot, Quarkus, and Micronaut make Java the gold standard — rich ecosystem, strong typing, proven at scale.

Core principles

Single Responsibility
Each service maps to a single business capability. User Service, Order Service, Payment Service — each with its own database.
Decentralized Data
No shared databases. Services own their data and expose it via APIs only. Enables polyglot persistence.
Design for Failure
Network calls fail. Implement circuit breakers, retries, timeouts, and bulkheads to prevent cascade failures.
Continuous Deployment
Each service is deployed independently via its own CI/CD pipeline. No coordinated releases across teams.

Service communication patterns

Synchronous — REST/gRPC
Used when the caller needs an immediate response. Implemented with Spring WebClient, RestTemplate, or OpenFeign. Suitable for read-heavy queries.
OpenFeignWebClientgRPC
Asynchronous — Events
Fire-and-forget via Kafka or RabbitMQ. Services react to domain events. Essential for saga orchestration and eventual consistency.
KafkaRabbitMQSpring Events

Data management

Database per service
Each service has its own DB. No shared schema. Choose the right DB for the job: PostgreSQL, MongoDB, Redis, Cassandra.
CQRS
Separate Command (write) and Query (read) models. Read-optimized views built from events. Scales independently.
Event Sourcing
Store state changes as a sequence of events. Enables full audit trail, time travel, and rebuilding projections.

Design Patterns

Essential patterns every Java microservice architect must master.

Circuit Breaker
intermediate
Wraps remote calls and trips open when failures exceed a threshold, preventing cascade failures. Implemented via Resilience4j. Has three states: CLOSED (normal), OPEN (failing fast), HALF-OPEN (testing recovery).
Resilience4j@CircuitBreakerFallback
Saga Pattern
advanced
Manages distributed transactions across services. Two variants: Choreography (services react to events) and Orchestration (central saga coordinator). Each step has a compensating transaction for rollback.
ChoreographyOrchestrationCompensation
API Gateway
intermediate
Single entry point for all clients. Handles cross-cutting concerns: auth, rate limiting, SSL termination, request routing, and response aggregation. Use Spring Cloud Gateway or Kong.
Spring Cloud GatewayRate LimitingAuth Filter
Service Discovery
intermediate
Services register themselves and discover each other dynamically. Eliminates hardcoded IPs. Client-side (Ribbon) or server-side (Nginx). Use Netflix Eureka or Consul.
EurekaConsulLoad Balancing
Outbox Pattern
advanced
Guarantees at-least-once event publishing. Write to local DB and outbox table in one transaction. A relay process polls the outbox and publishes to the message broker atomically.
TransactionalDebezium CDCReliability
Strangler Fig
beginner
Migrate from monolith to microservices incrementally. New features built as services. Traffic gradually shifted. Old monolith code retired as services mature. Low risk migration path.
MigrationIncrementalTraffic Routing

Code Labs

Annotated Java examples for the most critical microservice patterns.

Spring Boot Microservice — Order Service
A minimal production-ready service with layered architecture, JPA, and REST endpoint.
@SpringBootApplication
@EnableDiscoveryClient
public class OrderServiceApplication {
public static void main(String[] args) {
SpringApplication.run(OrderServiceApplication.class, args);
}
}

@RestController
@RequestMapping("/api/orders")
public class OrderController {
@Autowired private OrderService orderService;

@PostMapping
public ResponseEntity<Order> createOrder(@RequestBody OrderRequest req) {
Order order = orderService.createOrder(req);
return ResponseEntity.status(HttpStatus.CREATED).body(order);
}

@GetMapping("/{id}")
public ResponseEntity<Order> getOrder(@PathVariable Long id) {
return orderService.findById(id)
.map(ResponseEntity::ok)
.orElse(ResponseEntity.notFound().build());
}
}
Feign Client — Calling Inventory Service
Declarative HTTP client with built-in load balancing via Spring Cloud LoadBalancer.
@FeignClient(name = "inventory-service", fallback = InventoryFallback.class)
public interface InventoryClient {

@GetMapping("/api/inventory/{productId}")
InventoryResponse checkStock(@PathVariable String productId);

@PutMapping("/api/inventory/reserve")
ReservationResponse reserveItem(@RequestBody ReservationRequest req);
}

@Component
class InventoryFallback implements InventoryClient {
@Override
public InventoryResponse checkStock(String productId) {
// Return cached/default response on failure
return new InventoryResponse(productId, false, "Service unavailable");
}
}
Circuit Breaker with Resilience4j
Wraps service calls with circuit breaker, retry, and time limiter. Configure thresholds in application.yml.
@Service
public class PaymentService {
@Autowired private PaymentGatewayClient gateway;

@CircuitBreaker(name = "paymentGateway", fallbackMethod = "paymentFallback")
@Retry(name = "paymentGateway")
@TimeLimiter(name = "paymentGateway")
public CompletableFuture<PaymentResult> processPayment(PaymentRequest req) {
return CompletableFuture.supplyAsync(() -> gateway.charge(req));
}

public CompletableFuture<PaymentResult> paymentFallback(
PaymentRequest req, Throwable ex) {
log.warn("Payment gateway down, queuing: {}", ex.getMessage());
eventPublisher.publish(new PaymentDeferredEvent(req));
return CompletableFuture.completedFuture(
PaymentResult.deferred(req.getOrderId()));
}
}
Kafka Event Producer — Order Placed Event
Publish domain events to Kafka. Include correlation ID for distributed tracing across services.
@Service
public class OrderEventPublisher {
private final KafkaTemplate<String, OrderEvent> kafkaTemplate;
private static final String TOPIC = "order.placed";

public void publishOrderPlaced(Order order) {
OrderEvent event = OrderEvent.builder()
.eventId(UUID.randomUUID().toString())
.orderId(order.getId())
.userId(order.getUserId())
.totalAmount(order.getTotal())
.items(order.getItems())
.occurredAt(Instant.now())
.build();

Message<OrderEvent> msg = MessageBuilder
.withPayload(event)
.setHeader(KafkaHeaders.TOPIC, TOPIC)
.setHeader(KafkaHeaders.KEY, order.getId().toString())
.build();

kafkaTemplate.send(msg);
log.info("Published OrderPlaced event: {}", event.getEventId());
}
}
JWT Security in API Gateway
Validate JWT tokens at the gateway. Downstream services receive pre-validated user context via headers.
@Component
public class JwtAuthFilter extends AbstractGatewayFilterFactory<JwtAuthFilter.Config> {

@Override
public GatewayFilter apply(Config config) {
return (exchange, chain) -> {
String token = extractToken(exchange.getRequest());
if (token == null) return unauthorized(exchange);

return jwtValidator.validate(token)
.flatMap(claims -> {
ServerHttpRequest req = exchange.getRequest().mutate()
.header("X-User-Id", claims.getSubject())
.header("X-User-Roles", claims.get("roles"))
.build();
return chain.filter(exchange.mutate().request(req).build());
})
.onErrorResume(e -> unauthorized(exchange));
};
}
}

Tools & Stack

The modern Java microservice technology landscape — pick what fits your context.

Frameworks

★ popularSpring Boot
De facto standard. Rich ecosystem, auto-configuration, production-ready actuators. Pairs with Spring Cloud for full microservice stack.
Quarkus
Kubernetes-native. Sub-millisecond startup, low memory. Compiled to native with GraalVM. Great for serverless and sidecar patterns.
Micronaut
Compile-time DI — no reflection overhead. Fast startup, low memory. Built for cloud-native from the ground up.

Infrastructure

Docker & Kubernetes
Containerize each service with a multi-stage Dockerfile. Deploy on K8s with Deployment, Service, and ConfigMap manifests. Use Helm for templating.
