Part 1 — Iterative vs Recursive
AspectIterativeRecursive
MechanismUses loops (for/while) with explicit variablesFunction calls itself with a smaller input
MemoryO(1) — just a few variablesO(n) — one stack frame per call
SpeedGenerally faster — no function call overheadSlower — push/pop stack frames each call
Stack overflowCannot happenCan crash for large n (stack depth limit)
ReadabilityCan be complex for tree/divide-and-conquer problemsOften cleaner and closer to the math definition
Base caseLoop condition (i < n)Explicit if check inside the function
Best forSimple counting, accumulation, linear scanTrees, graphs, divide-and-conquer, mathematical definitions
Factorial4 lines, O(n) time O(1) space3 lines, O(n) time O(n) space
Fibonacci5 lines, O(n) time O(1) space3 lines, O(2ⁿ) time O(n) space — needs memoisation
Quick SortNeeds an explicit stack data structureNatural and elegant — the logic IS the recursion
The golden rule: use recursion when the problem is naturally recursive (trees, divide-and-conquer, mathematical induction). Use iteration when the problem is a simple count or scan. For Fibonacci specifically, always prefer the iterative version in production — the naive recursive is O(2ⁿ).

iterative_factorial.c
int factorial(int n) {
  int result = 1;
  for (int i=1; i<=n; i++)
    result *= i;
  return result;
}
recursive_factorial.c
int factorial(int n) {
  if (n <= 1) return 1;
  return n * factorial(n-1);
}
Both compute the same answer. The iterative version accumulates into result going up from 1 to n. The recursive version multiplies going down from n to 1 then back up. Same time complexity O(n), but iterative uses O(1) space while recursive uses O(n) stack frames.

iterative_fib.c
int fib(int n) {
  if (n <= 1) return n;
  int a=0, b=1, c;
  for(int i=2; i<=n; i++){
    c = a + b;
    a = b; b = c;
  }
  return b;
}
recursive_fib.c
int fib(int n) {
  if (n <= 1) return n;
  return fib(n-1) + fib(n-2);
}
// fib(7) = 25 calls
// fib(10) = 177 calls
// fib(20) = 21891 calls!
This is the clearest case where recursion hurts. The iterative version computes fib(n) in exactly n−1 steps. The recursive version recomputes the same subproblems over and over — fib(7) alone makes 25 calls. For fib(20) that's 21,891 calls vs 19 steps. Always use iterative or memoised for Fibonacci.
